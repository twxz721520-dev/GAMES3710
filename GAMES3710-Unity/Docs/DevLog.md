## 讨论与决策记录

### 2026-01-26 敌人AI系统设计

#### 讨论：敌人数量
- **问题**: 同时存在多个敌人还是单一敌人？
- **决策**: 同时仅1个敌人

#### 讨论：敌人移动方式
- **方案对比**:
  | 方案 | 优点 | 缺点 |
  |------|------|------|
  | NavMesh寻路 | 自动绕障碍、追逐灵活 | 需烘焙、动态障碍需额外处理 |
  | 预设路径点(Waypoint) | 完全可控、性能开销小 | 追逐时需配合其他方案 |
- **决策**: 混合使用
  - 巡逻阶段：Waypoint路径点
  - 追逐阶段：NavMesh实时寻路
  - 搜索阶段：NavMesh随机采样

#### 讨论：巡逻路线随机化
- **需求**: 敌人的固定路线分为几个阶段，每到一个固定路线的终点随机寻找下一个固定路线
- **决策**: 路径片段池设计，每个PatrolPath记录可连接的下一路径列表

#### 讨论：搜索阶段行为
- **方案对比**:
  | 方案 | 描述 |
  |------|------|
  | 基于丢失位置 | 以玩家最后可见位置为中心搜索附近预设点 |
  | 区域绑定搜索点 | 每个区域有专属搜索点池 |
- **决策**: 简化方案 - 以玩家最后可见位置为中心，NavMesh随机采样游荡
- **理由**: 不需要手动布置搜索点，实现简单，效果自然

#### 讨论：搜索阶段敌人行为细节
- **问题1**: 敌人搜索时会检查躲藏点吗？
- **决策**: 不检查，躲藏点绝对安全

- **问题2**: 搜索行为倾向？
- **决策**: 随机游荡一段时间

- **问题3**: 搜索阶段被发现后的行为？
- **决策**: 直接切换追逐，无渐进过程

#### 讨论：实现复杂度
- **原则**: 不使用过难的方案，在不牺牲扩展性的情况下选择简单而效果过得去的方案

---

### 2026-01-26 视线检测系统设计

#### 讨论：检测方式
- **方案对比**:
  | 方案 | 描述 | 复杂度 |
  |------|------|--------|
  | A. 简单锥形+射线 | FOV角度判断 + 单射线遮挡检测 | 低 |
  | B. 多射线采样 | 射向玩家多个点，任一可见即发现 | 中 |
  | C. 渐进发现 | 可见时间累积到阈值才触发追逐 | 中 |
- **决策**: 方案A（简单锥形+射线）
- **理由**: 够用且简单

#### 讨论：渐进发现机制
- **问题**: 敌人是否需要盯着玩家一段时间才确认？
- **决策**: 不需要，直接触发追逐

#### 讨论：扩展性预留
- **决策**: 预留声音吸引敌人的扩展性（暂不实现）

### 2026-01-26 躲藏点系统设计

#### 讨论：躲藏点类型
- **问题**: 固定交互物（柜子）还是遮挡物（桌子底下）？
- **决策**: 统一按键交互进入，即使是遮挡物也是按键进入

#### 讨论：躲藏时视角
- **理想方案**: 有限转动视角
- **决策**: 初期完全锁定，后续可扩展为有限转动
- **理由**: 降低初期实现复杂度

---

### 2026-01-26 资源管理系统设计

#### 讨论：San值机制
- **问题1**: 降低触发条件？
- **决策**: 时间自然流逝

- **问题2**: 低San值效果？
- **决策**: 后处理滤镜

- **问题3**: 归零后果？
- **决策**: 游戏结束

#### 讨论：电池/夜视系统
- **问题1**: 持续开启还是按需开关？
- **决策**: 按需开关

- **问题2**: 电池耗尽后夜视效果？
- **决策**: 直接关闭

- **问题3**: 电池充电方式？
- **决策**: 收集后自动充满

### 2026-01-26 游戏结构设计

#### 讨论：死亡与存档
- **被抓后果**: 游戏结束
- **存档机制**: 按关卡自动存档

#### 讨论：关卡设计
- **决策**: 三个关卡

| 关卡 | 类型 | 核心玩法 |
|------|------|----------|
| 1 | 探索 | 无敌人，纯收集解谜 |
| 2 | 潜行 | 室内巡游追逐 + 收集解锁 |
| 3 | 逃跑 | 高速追逐，选择路线避障 |

#### 讨论：关卡3详细设计
- **视角**: 第一人称
- **移动**: 自动向前跑
- **场景**: 预设路线与障碍物
- **操作**: 按键选择左/右转向或偏移
- **失败条件**:
  - 撞障碍物：减速，累计2次被追上→游戏结束
  - 进入绝路：直接游戏结束
- **设计意图**: 记忆正确路线 或 运气

#### 讨论：系统与关卡对应
| 系统 | 关卡1 | 关卡2 | 关卡3 |
|------|:-----:|:-----:|:-----:|
| 敌人AI | ✗ | ✓ | 特殊(追逐者) |
| San值 | ✓ | ✓ | ✗ |
| 电池/夜视 | ✓ | ✓ | ✗ |
| 锁钥解谜 | ✓ | ✓ | ✗ |
| 躲藏点 | ✗ | ✓ | ✗ |

#### 讨论：场景光照
- **决策**: 场景有明亮/黑暗区域分布，玩家需策略性使用夜视相机

### 2026-01-26 Waypoint结构设计

#### 讨论：两种方案
| 方案 | 描述 |
|------|------|
| A. 独立节点 | 每个Waypoint是独立GameObject，通过引用连接 |
| B. 路径为父物体 | PatrolPath下的子物体自动成为该路径的点 |

- **决策**: 方案B
- **理由**: 编辑器里拖拽调整更直观，片段管理清晰
